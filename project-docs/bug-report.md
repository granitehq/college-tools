# Bug Report

Captured from user-reported issues in the live spreadsheet. Descriptions and likely causes only — no fixes applied yet.

---

## Bug 1: Region not filled in on Colleges tab, and dropdown won't let it be corrected

**Status: Fixed in code.**

**Description:** On the `Colleges` tab, the Region column is not populated for any college, and attempting to select a value from the dropdown doesn't correct it.

**Likely cause:**
- `Region` is resolved via `optionalCollegeColumn_()` in `src/colleges.js` (`buildCollegesColumnMap_`, ~L62-90), unlike required columns (Name, City, State, etc.) which use `requireCol_()` and throw if the header text doesn't match exactly. If the live sheet's row-2 header for this column doesn't exactly match `"Region"` (extra whitespace, renamed, reordered, or an older template predating this column), `COL.REGION` silently resolves to `-1` and every fill/refill skips writing Region entirely (`fillCollegeRowCore`, ~L309, L419) with no error surfaced.
- `Region` is not in `PRESERVED_HEADERS` (~L92-107), so it is blanked on every refill and only repopulated if a fill path (`fillCollegeRowCore`, `fillRegionsAllRows`, or `repairEntireWorkbook`) actually completes. If a fill fails partway (e.g., no API match — see Bug 3) or the user edits State manually without re-running Fill, Region stays blank.
- The dropdown (`formatting.js`, `validateList`) restricts entries to the exact strings `Northeast/Midwest/South/West` with `setAllowInvalid(false)`. If a fill did partially succeed and wrote a value outside that set (or the column resolved to the wrong index due to the header mismatch above), manual correction via the dropdown would appear to "not take."
- The region-mapping logic itself (`Utils.getRegionForState`, `Config.REGION_MAP`) appears internally correct and is covered by passing tests, so the defect is most likely structural (header/column resolution or fill-path failure) rather than in the state→region mapping.
- Confirmed root cause: older/existing Colleges sheets predate the Region column, so it's simply absent from row 2 on those sheets — `optionalCollegeColumn_()` correctly resolves to `-1` for a genuinely missing column, and the dropdown was never attached because `validateList` also can't find a column that isn't there.

**Fix applied:** Added `ensureCollegesRegionColumn_()` in `src/colleges.js`, which detects a missing `Region` header, inserts the column immediately after `State`, writes the header label, and attaches the `Northeast/Midwest/South/West` dropdown validation. It's now called before header resolution in `fillCollegeRowCore`, `fillRegionsAllRows`, and `fillSelectedRows`, so any Fill action on an older sheet self-repairs the column first. Covered by regression tests in `test/regression-tests.js`.

**Priority:** High — a core categorization field is unusable and can't be manually recovered by the user.

## Bug 2: Fill leaves behind old notes from example/sample rows

**Description:** If the original example rows in the Colleges tab haven't been overwritten yet, filling in a real college on top of one can leave the sample row's old Notes text behind instead of writing fresh notes.

**Likely cause:**
- `isAutoStampNotes_()` in `src/colleges.js` (~L109-120) only treats a Notes cell as safe to overwrite if it's empty or matches the exact auto-stamp pattern the tool itself writes (`"X.Y.Z | ..."`, a version number and pipe). Any other text — including whatever placeholder/sample text ships in the template's example rows — is indistinguishable from genuine user notes and is preserved verbatim, since `Notes` is in `PRESERVED_HEADERS` and survives the row-clear step.
- Because the sample text was authored directly into the template (not generated by this codebase), it will never match the auto-stamp regex, so it's always treated as "user-entered" and kept on the first real fill.

**Priority:** Medium — cosmetic/confusing data leakage rather than functional breakage, but affects every fresh copy of the template on first use.

## Bug 3: Tracker sheets (Application Status Tracker, Application Timeline, Financial Aid Tracker, Campus Visit Tracker, etc.) show sample colleges instead of the primary list from Colleges

**Status: Fixed in code (both identified gaps addressed).**

**Description:** Suspected earlier as a row-offset bug (Colleges data starts row 3; other sheets start row 2), user believed this was fixed, but sample colleges are still showing instead of syncing from the Colleges tab.

**Likely cause:**
- The row-offset conversion itself (`getTrackerRowForCollegeRow_()` in `src/trackers.js`, ~L23-25: `Math.max(2, sourceRow - 1)`) is correctly implemented and consistently used by both `syncCollegeToTrackers` and `repairCollegeSync` — this specific offset bug appears to already be fixed at the code level.
- The real gap: sync to trackers only happens (a) inside `fillCollegeRowCore`, **after** a successful Scorecard API match — if the API lookup fails (`apiResult.ok === false`, ~L335-343), the function returns early before ever calling `syncCollegeToTrackers` (~L446-452), leaving that row's trackers untouched; or (b) via the explicit "Repair Entire Workbook" menu command (`repairCollegeSync`).
- So rows where the Fill action failed to match the API (typos, small/foreign schools not in Scorecard, name variants), or rows where the user typed a college name directly without ever running Fill, never get synced — their tracker rows keep showing whatever sample data shipped in the template. Running "Repair Entire Workbook" should resync all rows regardless of per-row fill history and is worth testing as a diagnostic.

**Fix applied:**
- `fillCollegeRowCore` (`src/colleges.js`) now calls `CollegeTools.Trackers.syncCollegeToTrackers()` with the typed/sanitized name on the no-match failure path too, not just after a successful API match — so trackers get the real name even when Scorecard can't find the school.
- Tracker setup (`src/trackers.js`, end of the setup flow) now automatically runs `repairCollegeSync({suppressAlert: true})` after creating/updating tracker sheets, and reports the synced row count in the completion alert — so running Setup/Refresh Trackers resyncs every row from Colleges regardless of per-row fill history, instead of relying on the user to separately run "Repair Entire Workbook."
- Covered by regression tests in `test/regression-tests.js` (tracker sync on failed lookup, and existing repair/sync alignment tests).

**Priority:** High — affects multiple tracker sheets and blocks the core cross-sheet workflow.

## Bug 4: Total Cost of Attendance and Net Price on Colleges tab seem low; no data year shown; Net Price doesn't reflect user's income

**Description:** Total cost of attendance appears lower than expected — possibly because Scorecard data is older than assumed. Net price also seems low and doesn't appear to account for the household income entered in the Personal Profile relative to Scorecard's income-bracket data.

**Likely cause:**
- `src/colleges.js` (~L360-364) reads `latest.cost.attendance.academic_year` and `latest.cost.avg_net_price.overall` from the Scorecard API. Scorecard's `latest.*` cost/earnings fields commonly lag 2-3 years behind the current admissions cycle, and this codebase never requests, stores, or displays which cohort year "latest" actually corresponds to — so there's no way for a user to know the figures are dated.
- Net Price uses `avg_net_price.overall`, a blended population-wide average across all students — not one of the Scorecard's income-bracket-specific net price fields (e.g., `net_price.*.by_income_level.*`). The code never requests or uses any income-bracket field. Meanwhile, the Personal Profile's `Family_Income` named range (`src/financial.js`, ~L32/L41) is only used downstream in Financial Aid Tracker ratio formulas (e.g., cost ÷ Family_Income) — never to select an income-appropriate net price from the API. So the displayed net price is the same for every household regardless of entered income, which can read as "too low" for lower-income families expecting a bracket-adjusted figure.

**Priority:** Medium — data is technically accurate per the API but likely to mislead users without context (year label, income-adjusted framing).

## Bug 5: Financial Aid Tracker — FAFSA/CSS deadlines and Total Cost of Attendance

**Description:** User assumed FAFSA deadline (and possibly CSS Profile deadline) might be treated as a single value applied to every college rather than being college-specific. Total Cost of Attendance also reappears on this tracker and may have the same "seems low" staleness issue as Bug 4.

**Likely cause:**
- No code path in `src/` writes a value into `FAFSA Deadline` or `CSS Deadline` for any college. `formatting.js` only applies date-format validation to these columns; `dashboard.js` only reads the corresponding `*Submitted (Y/N)` flags, not the deadlines; `trackers.js`'s `createOrUpdateFinAid` seeds formulas for other financial columns but explicitly leaves these two as manual fill-down fields. The Scorecard API fields requested (`Config.API_FIELDS`) contain no FAFSA/CSS deadline data at all — this information isn't in the dataset this tool consumes.
- Given that, these fields are 100% manual entry by design; there's no code mechanism that could propagate one college's deadline to all others. The reported symptom is more likely a spreadsheet data-entry artifact (e.g., fill-down or copy/paste of one row's dates across all rows) than a code defect — worth confirming with the user directly.
- Total Cost of Attendance on this tracker (`syncCollegeToTrackers` in `src/trackers.js`, ~L383-385) is written from the same `latest.cost.attendance.academic_year` value used on the Colleges tab, so it shares the exact same missing-year staleness issue as Bug 4, just duplicated onto this sheet.

**Priority:** Low for the deadlines (likely user data-entry, not a defect — needs confirmation); Medium for the duplicated Total Cost of Attendance year issue (same as Bug 4).
